## 🧠 判定ロジックの設計思想

### CSフォロー不足が判定されるパス
顧客は「CSのせいで解約した」とは言わない。以下の考え方でCSフォロー不足を検出する：

| 状況 | 考え方 |
|---|---|
| 利用なし（nashi）全般 | 在庫変動が止まっているのにCSが介入していなければフォロー不足 |
| 多忙・前向きになれない | CSが能動的にフォローしていれば防げた可能性がある |
| OB完了後に使いこなせない | アダプションフェーズのフォローが機能していない |
| 月額高いと感じる（利用なし） | 使えていないから高く感じる＝CSが価値訴求できていない |

### q_cs_nashi の判定基準（要運用後レビュー）
- **不十分の基準**：月2回以下 / 様子伺いのみ / 課題が記録されているのに次アクション未設定
- **十分の基準**：月1回以上 + 操作サポート・活用提案・課題解決の記録あり
- 確認場所：「Adaption_Schedule」シートの「進行中のアクション」列

## 🗂️ 解約理由カテゴリ（全25項目）

| カテゴリ | 項目 |
|---|---|
| A. 製品・サービス起因 | 機能への不満・不足 / 連携サービスへの不満・不足 / 開発優先度・スピードへの不満 / 他手段で代替可能 / 費用対効果 / サポートサービスの不足 |
| B. 顧客側の事情起因 | リソース不足 / ITリテラシーが低い / 低モチベーション / 引き継ぎ不備 / 予算不足・削減 / 上層部起因 / 従業員起因 / グループ会社等の事情 / ReCOREでは解決できない課題 |
| C. 自社（RECORE）起因 | 契約内容の不備 / 契約時の説明不足 / 営業担当の応対への不満・不足 / サクセスフォローの不足 / CS担当の応対への不満・不足 |
| D. 事業継続不可 | M&A退店・撤退 / 倒産・清算 / 未入金によりサービス停止 / アカウント整理 |
| E. その他 | その他（追加ヒアリング必須） |
